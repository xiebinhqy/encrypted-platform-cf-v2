## 1. 核心诊断

### 1.1 为什么 CF 上只有 4/10 分？

| 症状 | 根因 | 修复策略 | 预期提升 |
|------|------|---------|:--------:|
| 每次刷新都重新加载 | **前端无本地缓存**——Service Worker + IndexedDB 未被正确使用 | 修复 note-cache.js 在 modern 版的集成 | 6→8 分 |
| 仪表盘图表最慢 | D1 聚合查询慢 + 每次重新计算 | 前端计算结果缓存 + 后端预聚合 | 7 分 |
| 登录 1.5s | D1 冷启动 + Worker ismorphic 冷启动 | 配合缓存策略减少冷查询频率 | 1s 以内 |
| 编辑/保存 0.5s | API 请求 + 解密链路 | 乐观更新 + 后台同步 | 感觉即时 |

### 1.2 核心结论

```
本地流畅（10/10） ≠ CF 流畅（4/10）
原因不在代码质量，而在"每次刷新都重做一切"
你的 note-cache.js（IndexedDB缓存）+ sw.js（Service Worker）已经写了
但 modern 版前端没有正确使用它们
```

## 3. 阶段零：CF 性能急救（1周）

### 目标
不写一行新后端代码，通过修复前端缓存策略，将体验从 4/10 提升到 7/10。

### 需要修改的文件

| 文件 | 当前问题 | 修改内容 |
|------|---------|---------|
| `frontend/modern/js/services/data-loader.js` | 每次从 API 拉全量数据，解密后不缓存 | 集成 IndexedDB 缓存：先查缓存→有则直接渲染→后台静默刷新 |
| `frontend/modern/js/app.js` | 初始化时没有注册 Service Worker | 添加 `navigator.serviceWorker.register('/sw.js')` |
| `frontend/sw.js` | 可能没有被正确加载 | 检查 Service Worker 注册逻辑，添加静态资源缓存策略 |
| `frontend/shared/utils/note-cache.js` | 可能未被 modern 版调用 | 确认导出接口，接入 data-loader 的加载流程 |
| `frontend/modern/js/services/dashboard-updater.js` | 每次刷新重新聚合计算 | 缓存仪表盘聚合结果到 IndexedDB，增量更新 |

### 缓存策略设计

```
首次访问（无缓存）：
  API 请求 → 解密 → 渲染 → 写入 IndexedDB + sessionStorage

第二次访问（有缓存）：
  IndexedDB 读取 → 立即渲染（感觉即时）→ 后台 API 静默检查更新

被动更新：
  保存/编辑笔记 → 同时更新 IndexedDB 缓存 + 触发 API
  
Service Worker 缓存：
  /api/notes → 网络优先，缓存后备
  /api/categories → 网络优先，缓存后备
  /api/dashboard → 缓存优先，15分钟过期
  ／static/* → 缓存优先，版本化
```

### 预期效果

| 操作 | 当前 | 优化后 |
|------|:---:|:------:|
| 首次打开页面 | 4-6s | 4-6s（不变，可接受） |
| 刷新页面 | 4-6s | **<1s**（从 IndexedDB 读取） |
| 仪表盘加载 | 最慢 | **<2s**（缓存聚合结果） |
| 登录响应 | 1.5s | ~1s（减少后端压力） |
| 编辑笔记 | 0.5s 迟钝 | **感觉即时**（乐观更新） |

### 4.2 关键决策：Debian 服务器上的加密笔记用什么数据库？

| 方案 | 优点 | 缺点 | 推荐度 |
|:----:|:----:|:----:|:-----:|
| **A) 保持 better-sqlite3** | 无需改代码，立即部署 | 不适合多用户并发 | ⭐⭐⭐ (推荐) |
| **B) 迁移到 PostgreSQL** | 适合多用户 + 未来扩展 | 需要重写数据库层代码 | ⭐⭐ |
| **C) 双库并行** | SQLite 跑笔记, PG 跑其他 | 管理复杂度增加 | ⭐⭐⭐⭐ (推荐) |

**我的建议：方案 C**

```
服务器上的加密笔记 Docker 版 → better-sqlite3（无需改代码）
监控/提醒/交易系统 → PostgreSQL（新功能自然选用）
```

理由：
1. 你的服务器是你自己用，不是多用户高并发场景，SQLite 完全足够
2. 不修改现有加密笔记代码，直接部署
3. 新功能用 PostgreSQL，符合最佳实践

### "自动登录探测你的服务器"方案

关于你提到的"我给你 IP 和密码，你来自动查"，这个功能实现如下：

```
方式一：SSH 自动采集（推荐）
  ├── 在服务器上部署监控 Agent（Node.js 脚本）

方式二：Agent 自动发现
  ├── Agent 扫描本机 /etc/nginx/sites-enabled/
  ├── 扫描 Docker PS
  ├── 扫描 systemd 服务列表
  ├── 扫描监听端口
  └── 自动注册到监控系统 → 你只需要选择"开启/关闭监控"
```

### 关于 us.kg 域名

us.kg 是免费域名平台，目前**不支持 API 自动续期**。所以：
- **可以实现：** 自动检测过期时间 + 提前通知你去手动续期
- **不能实现：** 自动帮你续费（us.kg 无 API）
- **替代方案：** 对于有 API 的注册商（阿里云/GoDaddy/Namecheap），可以实现全自动续期

## 8. 数据库选型建议

### 最终建议

| 用途 | 数据库 | 方案 | 理由 |
|:----:|:------:|:----:|:------|
| 加密笔记 (CF) | D1 (SQLite) | 保持不变 | 无需改动，免费 |
| 加密笔记 (服务器) | better-sqlite3 | 保持不变 | 直接部署，无需改代码 |
| 监控系统 | PostgreSQL | 新建 | 时序数据查询性能好 |
| 过期提醒 | PostgreSQL | 新建 | 关系型数据，PostgreSQL 合适 |
| 交易系统 | PostgreSQL | 新建 | 事务安全，并发支持好 |

### PostgreSQL 安装方式

```yaml
# docker-compose.pg.yml
version: '3.8'
services:
  postgres:
    image: postgres:16-alpine
    container_name: postgres-main
    restart: unless-stopped
    ports:
      - "5432:5432"
    environment:
      POSTGRES_PASSWORD: ${PG_PASSWORD}
      POSTGRES_DB: platform
    volumes:
      - pg-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  pg-data:
```

## 10. 风险清单

| 风险 | 概率 | 影响 | 缓解措施 |
