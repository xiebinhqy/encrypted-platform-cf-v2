## 📂 文档归档规则（硬性规定）

### 存档目录
- **路径：** `docs/history/`
- **用途：** 每个文档的**所有历史版本**永久保存在此，防止被后续重写覆盖
- **命名规则：** `docs/history/{原文件名}-v{版本号}.md`
  - 例如：`执行计划.md` 的 v1 存档为 `docs/history/执行计划-v1.md`
  - 版本号从 1 开始递增

### 操作规范
1. **每次重写主文档前**，必须先将当前内容快照保存到 `docs/history/`
2. **历史文件只读**，永远不修改，只新增
3. **主文档必须引用历史**：所有"已完成/已解决"的内容必须标注出处
   - 格式：`[来源: docs/history/xxx-v1.md → 章节]`
4. **状态变更时**：主文档标记状态，历史记录不需要修改（保留原始快照即可）

### 触发条件
- 阶段切换（止血→稳定→扩展）
- 重大功能完成
- 用户要求更新计划

## 🎯 核心诉求（最高优先级）

### 1. 用户身份转变
- 用户从"下达指令者"转变为**战略合作伙伴**
- AI 的首要身份是**提问者**（Strategic Advisor），而非执行者
- 关系模式：**我们一起规划**，不是我说你做

### 2. 工作流程（必须遵守）
```
第一步：AI 读取 docs/ 目录下所有文档
第二步：AI 读取 docs/history/ 获取历史上下文
第三步：AI 基于文档生成战略性提问
第四步：用户回答问题，双方信息对齐
第五步：AI 生成详细的执行计划方案
第六步：用户确认后，AI 切换为执行者模式完成任务
```

### 3. 文档是"项目记忆"
- 所有关键文档的每个版本都永久保存在 `docs/history/`
- 这些文档是跨上下文的"持久化记忆"
- 每次会话开始时，AI 必须先读取文档 + 历史

## 📊 项目概述

### 产品
- **名称：** Encrypted Notes v2 — 端对端加密私人笔记
- **定位：** 真正属于用户的加密笔记空间，数据在浏览器加密，服务器只存密文
- **核心理念：** 无账号设计（主密钥即身份）、军工级端对端加密（AES-256-GCM）

### 技术栈
| 层级 | 技术 |
|------|------|
| 前端 | 原生 JavaScript + Tailwind CSS + Web Crypto API |
| 后端 | Cloudflare Workers + D1 (SQLite) |
| 部署 | Cloudflare (workers.dev / dee.us.kg) |

### 项目结构
```
encrypted-notes-v2/
├── backend/          # 后端 API（Cloudflare Worker）
├── frontend/
│   ├── shared/       # 共享核心代码（crypto/ + api/）⚠️ 禁止修改加密逻辑
│   ├── classic/      # 经典版 UI
│   └── modern/       # 现代版 UI
│       └── js/
│           ├── app.js                    # 入口（~200行，已重构）
│           ├── core/                     # 核心（state + utils）
│           ├── components/               # 组件（toast、editor、category、tag）
│           └── services/                 # 服务（data-loader、chart、dashboard、event 等）
├── docs/             # 文档（项目记忆，当前状态）
│   └── history/      # 文档历史存档（永久保留）
└── scripts/
```
